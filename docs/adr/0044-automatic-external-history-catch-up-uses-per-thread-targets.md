# Automatic External History Catch-Up Uses Per-Thread Targets

Status: accepted

XiaoBa keeps `future-only` as the default External History Mode and adds an opt-in, durable `catch-up` mode per provider. When catch-up is enabled, the existing Distillation Heartbeat automatically drains bounded historical work. Operators select a policy once; they do not select topics, construct ranges, or trigger each batch.

The consistency boundary is one external thread, not an entire provider catalog. xURL exposes ordered thread timelines but no atomic provider-wide snapshot. The Runtime therefore records an immutable Catch-Up Target for each discovered in-scope thread from its stable completed-event position and cumulative prefix digest. Mutable cursor, progress, generation-membership, and lifecycle fields remain separate resource state. Historical events up to the target enter the ordinary external admission path, while later events continue through the continuous lane without extending the target.

The Runtime does not mirror a complete provider manifest. It expands the official xURL catalog limit over bounded wakes until one observation returns fewer resources than requested, and blocks rather than claiming completion if configured limits cannot prove that condition. It persists only the pass generation, requested limit, scope fingerprint, per-resource targets, cursors, and aggregate progress. Each resource records its latest observed generation, so a provider is caught up only after a complete pass and terminal resolution of every open in-scope target.

Historical evidence uses the existing Distillation Unit, Learning Episode, Evidence Capsule, Author, Verifier, and promotion pipeline. An episode admitted by catch-up remains `historical-pending` until its source thread reaches the fixed target. The review budget rotates durably among due retries, internal or live episodes, and ready historical episodes so a large history cannot starve ordinary learning or recovery.

Scheduling stays inside the existing Heartbeat, provider locks, bounded external read pool, and External Admission Coordinator. Internal work remains first, continuous work remains timely, and historical backlog receives one global bounded catch-up quantum per eligible discovery wake. A quantum advances catalog inventory, target stability sampling, or one evidence page, so preparation cannot starve behind already-ready work. Providers share quanta through durable round-robin; unused budget remains work-conserving. External reads may overlap, but durable Episode, Capsule, provenance, and cursor writes remain serialized.

Changing a provider from `catch-up` to `future-only` pauses unfinished targets without deleting progress. Re-enabling catch-up resumes the same targets. Scope narrowing pauses out-of-scope targets, while scope expansion baselines newly included resources in `future-only` mode and starts them from their earliest canonical event in `catch-up` mode. An explicit `rebaseline --skip-to-now` operation is the only normal way to abandon unfinished history; it requires future-only mode and writes durable skip tombstones and an operator audit.

Explicit bounded backfill remains available for repair, replay, and precise ranges. It does not become the scheduler for ordinary historical learning. A quarantined event blocks only its resource target, while a protocol failure or integrity conflict preserves the existing provider-level pause because normalized identity is no longer trustworthy. Other providers continue, dependent episodes remain pending, and the affected provider reports attention required until operator recovery.

This decision avoids a second scheduler, a historical-only reviewer, a full transcript mirror, and a false provider-wide snapshot guarantee. Thread targets guarantee that fixed historical evidence is admitted before review, but they do not expand the existing bounded Source Ref Window into arbitrary transcript search. Skill Evolution remains responsible for revising a capability when independent or semantically distant evidence becomes reviewable later.
